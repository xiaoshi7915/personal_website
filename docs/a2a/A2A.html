A2A协议技术深度研究
代理互联网的HTTP

探索Google于2025年推出的开放标准，如何通过标准化的通信协议打破AI代理间的孤岛效应， 构建一个互联互通、协作共赢的智能体生态系统。

2025年6月 50+行业伙伴 开源标准

执行摘要

A2A（Agent2Agent）协议是Google于2025年4月联合50余家行业伙伴推出的一项开放标准， 旨在成为"代理互联网的HTTP"。它通过定义一套基于现有Web技术（如HTTP、JSON-RPC 2.0、SSE）的通用通信规范， 打破不同AI代理（Agent）之间的技术壁垒，解决长期存在的"孤岛效应"。 [103]

A2A协议的核心组件包括用于能力发现的Agent Card、用于管理复杂协作的Task模型、 用于多模态信息交换的Message，以及用于交付结果的Artifact。 该协议由Linux基金会托管，确保了其开放与中立。 尽管面临性能、安全和标准化等挑战，A2A已在企业自动化、金融、医疗等领域展现出巨大潜力， 并获得了微软、Salesforce等巨头的支持，正朝着构建更复杂、更自主的多智能体生态系统方向发展。

A2A协议核心概念与原理

作为"代理互联网的HTTP"的架构设计理念

架构设计理念

A2A协议最核心的设计理念，就是将自己定位为"代理互联网（Agentic Internet）的HTTP" [113]。 正如HTTP协议通过提供一个简单、统一的请求-响应模型，将全球数以亿计的信息系统连接成一个庞大的万维网， A2A协议也希望通过提供一个标准化的通信框架，将未来数以亿计的AI代理连接成一个能够自主协作的智能网络。

开放标准与通用语言定位

A2A协议从诞生之初就被设计为一个开放标准，其规范以Apache 2.0许可证开源， 并交由Linux基金会进行中立治理[124]。

核心特征

  • • 开放标准，而非私有协议
  • • 基于现有的、被广泛接受的技术
  • • 足够简单和通用，适应复杂场景

基于现有Web标准的协议栈

HTTP(S) - 传输层协议

互联网最通用、最受支持的协议

JSON-RPC 2.0 - 数据交换格式

轻量级、语言无关的RPC协议

SSE - 实时通信

服务器推送技术，支持长任务

核心架构组件

Agent Card

智能体的"数字名片"或"能力说明书"，发布在 /.well-known/agent.json [100]

包含信息：身份描述、服务端点、能力声明、技能列表、认证要求

Task

管理复杂、长周期协作的核心抽象，定义任务完整生命周期 [33]

状态管理：submitted → working → completed/failed

Message

智能体之间信息交换的基本单元，支持多模态内容 [17]

内容类型：文本、文件、结构化数据、UI元素

Artifact

交付任务最终成果的核心组件，封装产出结果 [55]

结构：标准化、可扩展的结果交付方式

五大核心设计原则

拥抱Agent原生能力

支持和增强智能体的自主性、推理能力和工具使用能力 [3]

基于现有标准

建立在HTTP、JSON-RPC、SSE等成熟Web标准之上 [57]

默认安全

支持企业级的身份验证和授权机制 [57]

支持长任务

管理需要很长时间才能完成的任务，支持异步通信 [3]

模态无关

支持文本、图像、音频、视频等多种数据模态 [3]

实现方式与技术细节

基于现有Web标准的技术栈选择与实现

关键通信协议

HTTP/gRPC

A2A协议在传输层主要支持HTTP(S)和gRPC两种协议，HTTP为默认选项， gRPC作为高性能替代方案[114]

JSON-RPC 2.0

轻量级、语言无关的RPC协议，提供清晰的请求-响应结构 [114]

SSE

基于HTTP的服务器推送技术，支持长任务的实时进度反馈 [55]

技术选择考量

简单性与易用性

JSON-RPC规范简单，核心概念清晰，易于理解和实现

语言无关性

JSON被几乎所有现代编程语言原生支持，促进异构系统互操作

现有生态集成

充分利用HTTP成熟生态，包括状态码、头部、缓存机制等

多模态支持的实现机制

通信模式与交互流程

通信模式 技术实现 适用场景 优缺点
同步请求/响应 HTTP POST + JSON-RPC 简单查询、快速任务 简单、易于实现，但效率较低
流式传输 Server-Sent Events (SSE) 实时更新、增量结果 实时性好，用户体验佳
异步推送通知 Webhook回调 长期任务、异步处理 系统解耦，但实现相对复杂

Python SDK 使用示例

from a2a_sdk import Agent, Task, Message

# 创建Agent实例
agent = Agent(
    name="数据分析助手",
    endpoint="https://api.example.com/a2a",
    auth_token="your-token"
)

# 创建任务
task = agent.create_task(
    task_type="data_analysis",
    description="分析销售数据",
    input_data={"source": "sales_db", "period": "Q1"}
)

# 发送消息
message = Message(
    to="agent://report-generator",
    content="请生成报告",
    attachments=[{"url": "https://...", "type": "file"}]
)
agent.send_message(message)

# 获取任务状态
status = agent.get_task_status(task.id)
print(f"任务状态: {status.state}")

协议技术细节

HTTP端点规范

Agent Card发现端点：

GET /.well-known/agent.json

任务管理端点：

  • • POST /a2a/tasks - 创建新任务
  • • GET /a2a/tasks/{task_id} - 获取任务状态
  • • PUT /a2a/tasks/{task_id} - 更新任务
  • • DELETE /a2a/tasks/{task_id} - 取消任务

消息交换端点：

  • • POST /a2a/messages - 发送消息
  • • GET /a2a/messages/{message_id} - 获取消息
  • • GET /a2a/messages/stream - SSE流式接收消息

状态码规范

状态码 含义 使用场景
200 OK 请求成功 同步请求成功响应
202 Accepted 请求已接受 异步任务已创建
400 Bad Request 请求格式错误 JSON-RPC格式错误或参数无效
401 Unauthorized 未授权 缺少或无效的认证令牌
404 Not Found 资源不存在 任务或消息ID不存在
500 Internal Server Error 服务器错误 代理执行任务时发生错误

实际应用中的技术挑战与解决方案

安全性挑战

身份验证、授权和数据加密需求

解决方案：OAuth 2.0、API Key、mTLS、HTTPS加密

异构协作挑战

不同智能体间的语义互操作性

解决方案：标准化任务定义、JSON Schema、社区规范

性能优化

高并发、低延迟场景的性能需求

解决方案：异步处理、负载均衡、缓存策略、gRPC优化

A2A协议的优缺点分析

推动AI代理生态繁荣的双刃剑

优点：推动AI代理生态的繁荣

高度的互操作性与集成性

成功解决"孤岛效应"，不同厂商、技术栈的智能体能够无缝协同工作 [72]

基于开放标准，易于实现

建立在成熟Web标准之上，降低学习和实现门槛 [57]

强大的多模态交互支持

灵活支持文本、图像、文件、结构化数据等多种模态 [55]

默认安全的通信机制

支持OAuth 2.0、API密钥等企业级安全机制 [57]

缺点与面临的挑战

性能瓶颈

基于HTTP的潜在延迟，在高频、低延迟场景可能不是最佳选择 [56]

生态系统碎片化风险

不同厂商可能产生各自的扩展或修改，导致不兼容的"方言"

复杂任务编排挑战

缺乏标准化的方式来描述和编排复杂的多步骤工作流
